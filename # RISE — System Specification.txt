## 1. Application Overview

**RISE** is a personal life management application. Full name: **RISE ŌĆö My Organized Hub for Everything**.

It tracks actions, targets, visions, finances, rhythms, professional CRM, relationships, reviews, journal entries, and documents. It includes an AI assistant powered by Google Gemini.

- **Currency:** AED (UAE Dirham)
- **Region:** UAE-specific (banks, emirates, visa statuses)
- **User:** Single-user app with Google OAuth

## 3. Terminology

RISE enforces a strict hierarchy of terms. These names MUST be used in all UI labels, headings, buttons, descriptions, and toast messages.

| Term | Meaning | Old term (never use in UI) |
|------|---------|---------------------------|
| **Realm** | High-level life domain | Life Area, LifeArea |
| **Target** | Outcome or objective inside a Realm | Project |
| **Action** | Small, concrete step to complete a Target | Task |
| **Rhythm** | Recurring behavior or routine | Habit |

**Hierarchy:** Realm -> Target -> Action -> Rhythm

**Rules:**
- Every Target belongs to exactly one Realm
- Every Action belongs to exactly one Target (optional) and one Realm
- Every Rhythm is linked to a category
- Internal TypeScript interfaces (`Task`, `Project`, `Habit`) and Firestore collection names (`tasks`, `projects`, `habits`) remain unchanged for backward compatibility

## 6. Authentication

### 6.1 Client-Side Auth

**Provider:** `components/providers/AuthProvider.tsx`
- Firebase `onAuthStateChanged` listener
- Exposes: `user`, `loading`, `signOut`, `signInWithGoogle`
- `useAuth()` hook for accessing auth context

**Login flow:**
1. User visits `/login`
2. Clicks "Sign in with Google"
3. Firebase `GoogleAuthProvider` + `signInWithPopup`
4. On success: redirect to `/` (dashboard)
5. `(main)/layout.tsx` checks auth ŌĆö redirects to `/login` if no user

**Loading state:** Centered pulsing RISE logo

### 6.2 Server-Side Auth

**Module:** `lib/verify-auth.ts`
- Extracts Bearer token from `Authorization` header
- Verifies via Firebase `identitytoolkit.googleapis.com` REST API
- Returns `userId` (localId) or throws

Used by: `/api/chat`, `/api/transcribe`, `/api/ai-tip`

## 8. Firestore Collections

| Collection | Interface | Notes |
|-----------|-----------|-------|
| `tasks` | Task | Actions |
| `projects` | Project | Targets |
| `labels` | Label | Tags for actions |
| `goals` | Goal | Visions |
| `milestones` | Milestone | Vision milestones |
| `goal-actions` | GoalAction | Steps for visions |
| `habits` | Habit | Rhythms |
| `pomodoro-sessions` | PomodoroSession | Timer sessions |
| `leads` | Lead | CRM leads |
| `deals` | Deal | CRM deals |
| `transactions` | Transaction | Income/expense |
| `budgets` | Budget | Monthly limits |
| `debts` | Debt | Debt tracking |
| `connections` | Connection | Relationships |
| `reviews` | Review | GPS reviews |
| `journal-entries` | Journal | Journal entries |
| `documents` | RiseDocument | Document metadata |
| `chat-messages` | ChatMessage | AI chat history (synced across devices) |

**Query pattern:** All queries filter by `userId == currentUser.uid`. No composite indexes ŌĆö sorting is client-side to avoid index requirements.

**Cross-device sync:** Firestore is initialized with `memoryLocalCache()` (not `persistentLocalCache`). The IndexedDB-backed cache is intentionally **disabled** so every device reads the same authoritative server state via real-time `onSnapshot` listeners. Trade-off: no offline reads, but no cache divergence between mobile and PC. See `lib/firebase.ts` for the rationale and the upstream firebase-js-sdk issues this works around.

## 9. Pages ŌĆö Detailed Specification

### 9.1 Dashboard (Home)

**File:** `app/(main)/page.tsx`
**Path:** `/`

**Sections (top to bottom):**

1. **Dynamic Greeting**
   - Time-based: "Good morning/afternoon/evening"
   - Shows user's first name from Google account (no hardcoded fallback ŌĆö nothing shown if name unavailable)
   - Date + time on one line: "Tue, Apr 7, 2026 ┬Ę 10:30 AM" (clock updates every 60s)

2. **Quick Stats** (2x2 grid on mobile, 4-col on desktop)
   - Done today: count of tasks completed today (green)
   - Active targets: count of incomplete goals (blue)
   - Avg streak: average of all active rhythm streaks (orange)
   - Surplus/Deficit: monthly income - expenses (green if positive, red if negative, AED)

3. **Winner's Mindset** (collapsible)
   - 37 hardcoded affirmation strings
   - Shuffled randomly on page load
   - Collapsed by default, click to expand

4. **Today's Focus**
   - Max 3 tasks where `isMyDay === true`
   - Sorted: incomplete first
   - Uses TaskCard component
   - Empty: "No focus actions. Mark actions as My Day."

5. **Be Consistent** (Rhythms module)
   - Shows ALL pending rhythms for today (status === 'pending')
   - Sorted by time (earliest first), then alphabetically
   - Each rhythm shows: name, time (AM/PM), Done button (green), Failed button (red)
   - First 5 visible, expandable dropdown "Show X more" for rest
   - Marking Done or Failed removes rhythm from this list (real-time)
   - "View all rhythms" link to `/wellness`
   - Empty: "All rhythms completed for today!"

6. **Get Things Done**
   - Up to 5 tasks where `isMyDay || dueDate === today`
   - Sorted: incomplete first
   - Uses TaskCard component
   - "View all actions" link to `/tasks`

7. **Target Progress**
   - Top 3 goals sorted by progress descending
   - Each: title, area badge, progress %, progress bar (blue gradient)
   - "View all targets" link to `/goals`
   - Only shown if goals exist

### 9.2 Actions (Tasks)

**File:** `app/(main)/tasks/page.tsx`
**Path:** `/tasks`

**Header:** "Actions" + "New Action" button (opens TaskModal)

**5-Tab Navigation:**

| Tab | Label | Content |
|-----|-------|---------|
| 1 | Today | Overdue actions (red label) + actions due today |
| 2 | Inbox | Actions without a Target (no projectId) |
| 3 | Upcoming | Actions with future due dates |
| 4 | Completed | Actions completed in the last 7 days |
| 5 | Targets | Targets grouped by Realm, with progress bar (done/total) |

> GTD and Quadrant tabs have been **removed**. Only these 5 tabs exist.

**Completed-tab cutoff:** The Completed tab only renders actions whose `completedAt` is within the last 7 days. Older completed actions are **hidden from view** but remain in Firestore ŌĆö nothing is auto-deleted. (Previous versions deleted them on page load, which caused cross-device data loss when one device cleaned up before another synced.)

**TaskCard behavior:**
- Left: priority-colored border (P1=red, P2=amber, P3=blue, P4=gray)
- Completion toggle circle (colored by priority)

## 11. API Routes

### 11.1 POST `/api/chat`

**Purpose:** AI chat with Gemini
**Auth:** Bearer token (Firebase ID token)
**Rate limit:** 30 requests/minute/user
**Request:** `{ message: string, context: string, history: ChatMessage[] }`
**Response:** `{ reply: string }` or `{ error: string }`

**System prompt includes:**
- RISE terminology explanation
- User's full data context (sanitized)
- Instructions: concise, actionable, AED currency, UAE context

**Model:** Gemini 2.5 Flash via @google/genai

### 11.2 POST `/api/transcribe`

**Purpose:** Speech-to-text with transcript cleanup
**Auth:** Bearer token (Firebase ID token)
**Rate limit:** 30 requests/minute/user (shared with chat)
**Request:** FormData with `audio` file field
**Max file size:** 10 MB
**Supported formats:** webm, wav, mp4, m4a
**Response:** `{ rawTranscript: string, cleanedTranscript: string }` or `{ error: string }`

**Process:**
1. Raw transcription: Gemini multimodal with audio (preserves all fillers/stutters)
2. Cleanup: second Gemini call removes fillers (um, uh, like, you know), stutters, repetitions, false starts; fixes grammar; keeps meaning

### 11.3 POST `/api/ai-tip`

**Purpose:** Daily motivational tip
**Auth:** Bearer token (Firebase ID token)
**Rate limit:** 10 requests/day/user
**Request:** `{ context: string }`
**Response:** `{ tip: string }` or `{ error: string }`
**Fallback:** "Start your day with your most important action. Momentum builds success."

## 12. Hooks

### 12.1 useVoiceRecorder (`hooks/useVoiceRecorder.ts`)

**Purpose:** Capture microphone audio via MediaRecorder API

**Returns:**
```typescript
{
  state: {
    isRecording: boolean
    isPaused: boolean
    duration: number        // seconds elapsed
    error: string | null
  }
  startRecording(): Promise<void>
  stopRecording(): Promise<Blob | null>
  cancelRecording(): void
}
```

**Audio config:** echoCancellation, noiseSuppression, sampleRate 16000
**Format preference:** webm;codecs=opus > webm > mp4
**Data collection:** 250ms chunks
**Duration timer:** 1-second interval updates
**Errors:** Permission denied, device access failure ŌĆö descriptive messages

### 12.2 useCollection (`lib/firestore.ts`)

**Purpose:** Real-time Firestore collection subscription

**Signature:**
```typescript
useCollection<T>(
  collectionName: string,
  userId: string | undefined,
  orderField?: string,        // default 'createdAt'
  limitCount?: number,
  serverOrder?: boolean,      // default false (client-side sort)
  startAfterValue?: any,
  orderDirection?: 'asc' | 'desc'  // default 'desc'
): { data: T[], loading: boolean, error: string | null }
```

**Default mode (serverOrder=false):** queries by userId only (no composite indexes), sorts client-side
**Server mode (serverOrder=true):** uses Firestore orderBy (requires composite index)

### 12.3 useAuth (`components/providers/AuthProvider.tsx`)

**Returns:** `{ user: User | null, loading: boolean, signOut: () => void, signInWithGoogle: () => Promise<void> }`

## 15. Security & Privacy

### 15.1 Firestore Rules

```
All reads/writes require:
  - request.auth != null
  - resource.data.userId == request.auth.uid (for reads/updates)
  - request.resource.data.userId == request.auth.uid (for creates)
```

Complete per-user data isolation. No cross-user access.

### 15.2 Storage Rules

```
/users/{userId}/{allPaths=**}
  - read, write: request.auth.uid == userId
```

Files scoped to user directories only.

### 15.3 API Security

- All API routes verify Firebase ID token
- Rate limiting per user (sliding window)
- PII sanitization before sending data to Gemini
- Generic error messages to client (no API internals leaked)
- No CORS issues (same-origin API routes)

### 15.4 Client Security

- No secrets in client code (GEMINI_API_KEY is server-only)
- `NEXT_PUBLIC_*` prefix only for Firebase config (public by design)
- Auth state managed via Firebase onAuthStateChanged
- ID token refreshed automatically by Firebase SDK

### 16.2 Service Worker (`app/sw.ts`)

- Serwist-based service worker
- Precaches static assets from build manifest
- `skipWaiting` + `clientsClaim` for immediate activation
- Navigation preload enabled
- **Custom runtimeCaching order** (critical for cross-device sync):
  1. RSC prefetch requests ŌåÆ `NetworkFirst` (cacheName: `pages-rsc-prefetch`)
  2. RSC requests ŌåÆ `NetworkFirst` (cacheName: `pages-rsc`)
  3. Page navigations ŌåÆ `NetworkFirst` (cacheName: `pages`)
  4. `/api/*` routes ŌåÆ `NetworkOnly`
  5. `...defaultCache` (handles content-hashed JS chunks, images, fonts)
- The custom rules are registered **before** `defaultCache`, overriding its default `StaleWhileRevalidate` for pages. This guarantees that page HTML and API responses are never served from a stale cache, so mobile and PC always render the same shell after a deployment.

| Variable | Scope | Purpose |
|----------|-------|---------|
| `NEXT_PUBLIC_FIREBASE_API_KEY` | Client + Server | Firebase API key |
| `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN` | Client + Server | Firebase auth domain |
| `NEXT_PUBLIC_FIREBASE_PROJECT_ID` | Client + Server | Firebase project ID |
| `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` | Client + Server | Firebase storage bucket |
| `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID` | Client + Server | Firebase messaging sender |
| `NEXT_PUBLIC_FIREBASE_APP_ID` | Client + Server | Firebase app ID |
| `GEMINI_API_KEY` | Server only | Google Gemini API key |
| `NEXT_PUBLIC_APP_URL` | Client + Server | App base URL (default: http://localhost:3000) |

**Notes:**
- `NEXT_PUBLIC_*` variables are exposed to the browser ŌĆö Firebase config is public by design
- `GEMINI_API_KEY` is server-only ŌĆö never sent to client
- Missing Firebase vars: app logs warning, Firestore operations fail gracefully
- Missing Gemini key: AI endpoints return 500 with "No API key configured"
